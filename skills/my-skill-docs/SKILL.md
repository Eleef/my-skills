# 文档与归档（Docs）

保证项目文档与当前实现或阶段成果保持一致，并让后续协作可追踪。

## 工作原则
1. 仅处理文档同步、归档、索引维护、状态记录、验收留痕。
2. 不重新做需求澄清；若需求本身未稳定，转回 `my-skill-specify`。
3. 不重新做方案设计；若方案未定，转回 `my-skill-plan`。
4. 默认更新最少必要文档，不为小任务制造过多文档。
5. 若仓库已有文档结构，优先遵循现有结构；不要因技能规范强行改造目录。
6. 创建新目录或新文件前，优先确认这是当前任务所必需；若影响较大，应先征得用户同意。

## 启动检查
优先读取以下文件；缺失则按降级模式处理：
- `docs/00_project_index.md`
- `docs/dev_status/active_task.md`
- `docs/guide/conventions.md`
- `spec/` 下与当前任务相关的文件

## 降级模式
- 若关键文件当前无法访问，明确说明“当前无法访问”。
- 一次性列出需要补充的文件清单。
- 仅给最小可行建议，并标注假设与风险。

## 工作流
1. 判断本次任务属于哪类文档动作：
   - 文档同步
   - spec -> docs 归档(通常是一个大阶段且需要用户确认)
   - 验证记录
   - 暂停记录
   - 结项记录
2. 找到最少必要更新点：
   - 索引、active_task、history_log、todo、设计文档、产品文档等。
3. 执行更新：
   - 记录已确认事实，不编造成果。
   - 验证状态统一使用 `UNVERIFIED / BROKEN / VERIFIED`。
4. 若是归档：
   - 将稳定成果从 `spec/` 映射到 `docs/`。
   - 保持索引可追踪。
5. 若是暂停或结项：
   - 只记录高价值上下文：当前目标、已完成、未完成、阻塞项、下一步动作。

## 验证状态约定
- `UNVERIFIED`：未执行测试或未验证。
- `BROKEN`：已验证失败，需附修复方向。
- `VERIFIED`：已验证通过，附命令与关键结果摘要。

## 标准输出骨架

### 更新的文档
- `path/to/file`: ...

### 验证记录
- 状态：
- 命令：
- 摘要：

### 遗留事项
- ...

## 切换建议
- 若文档同步过程中发现需求仍不稳定，切回 `my-skill-specify`。
- 若文档同步过程中发现方案尚未确认，切回 `my-skill-plan`。

## 安全与真实性
- 不记录、不输出真实密钥、Token、私钥或生产数据；示例使用占位符。
- 未经用户同意，不执行 git 提交；若环境不支持 git，仅提供中文 commit message、建议命令序列与变更文件清单。
