# 编码计划（Planning）

## 概览
把已澄清的需求转为可执行的实现计划与设计草案，确保可交付、可验证、可追踪。

## 工作流（按顺序执行）
1. 确认需求已清晰；若仍存在歧义，提示切回 my-skill-specify 技能进行澄清。
2. 先做可访问性判断：无法读取的文件必须说明“当前无法访问”，进入降级模式：
   - 一次性列出需要用户补充的文件/片段清单。
   - 在现有信息基础上给出最小可行建议（MVP），并显式标注假设与风险。
   - 严禁编造文件内容、目录结构、配置、运行结果或项目历史。
3. 选择 DESIGN 或 IMPLEMENTING 语境并遵守边界：
   - DESIGN：方案对比、接口/数据模型草案、风险与验收标准。
   - IMPLEMENTING：变更列表（代码+文档）、关键片段、迁移/兼容说明。
4. 生成计划与设计输出，必须按交付格式组织：
   1) ✅ 变更摘要（Why / What）
   2) 📄 文档变更清单（文件路径 + 章节/要点）
   3) 🧩 代码变更清单（文件路径 + 关键片段或 diff）
   4) 🧪 验证方式（命令 + 预期）与状态（VERIFIED/BROKEN/UNVERIFIED）
   5) ⚠️ 风险、兼容性与回滚策略
5. 将计划书落盘到 `spec/`：
   - 推荐路径：`spec/features/<feature-key>/05_plan.md`，并在 `spec/00_index.md` 记录索引。
   - 若 `spec/` 或功能目录不存在，先创建最小结构后再写入。
6. 计划中必须体现文档同步与测试策略：
   - 代码变更必须同步更新对应文档。
   - 正式测试放在 `tests/`，临时验证脚本放在 `scripts/scratch/`。
   - 测试命令参考 `docs/guide/testing.md` 或 `docs/guide/development.md`；若缺失且技术栈不明，提示切回 my-skill-specify 技能进行澄清。
7. 未运行测试必须标注 UNVERIFIED；失败标注 BROKEN 并给出修复计划；通过标注 VERIFIED（附命令与关键输出摘要）。
8. 若计划包含技术设计文档，请参考本技能末尾的“技术设计模板”部分。

## 输出要求
- 最终对用户输出中文（术语/专有名词可保留英文）。
- 计划内容要可执行、可验证，并可与后续实现/验收对齐。

## 附录：计划细节与设计模板

### 1. 常见拆解策略
- **里程碑拆解**（推荐）：
  - MVP：核心链路打通，无非核心特性。
  - P1：错误处理、边界情况、完整验证。
  - P2：性能优化、可观测性、扩展特性。
- **模块拆解**：按功能域、微服务或分层（API -> Service -> DAO）拆分。
- **风险驱动**：优先实现高风险、高不确定性模块（Spike Solution）。

### 2. 风险与回滚提示
- **数据兼容性**：Schema 变更是否向后兼容？是否需要迁移脚本？
- **接口变更**：是否破坏现有 Client？版本控制策略？
- **性能影响**：是否引入慢查询、N+1 问题或内存泄漏风险？
- **回滚策略**：明确可由 Feature Flag 关停，还是必须代码回滚？数据污染如何清理？

### 3. 技术设计模板（Design Doc）
若任务复杂度较高（>5文件或涉及核心架构），建议输出独立设计文档：

#### 标题：[Feature Name] Technical Design
- **概述 (Overview)**：一句话描述变更目标与范围。
- **方案对比 (Alternatives)**：
  - 方案 A（推荐）：优点、缺点、成本。
  - 方案 B：...
- **架构/流程 (Architecture/Flow)**：
  - 关键时序图或数据流向（Mermaid）。
  - 组件交互说明。
- **数据模型 (Data Model)**：
  - 新增/修改的表结构、字段、枚举。
  - 配置项定义。
- **接口设计 (API Design)**：
  - Route / Method / Request / Response 关键定义。
- **安全与权限 (Security)**：鉴权、脱敏、Rate Limit。
