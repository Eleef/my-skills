# 编码计划（Planning）

将已澄清的需求转成可执行、可验证、可交付的实施方案。

## 工作原则
1. 仅在需求已基本清晰时使用；若仍有核心歧义，切回 `my-skill-specify`。
2. 本技能用于约束计划内容与落盘方式，不替代 Claude Code 原生 `EnterPlanMode / ExitPlanMode`。
3. 默认先在对话中输出计划；仅在用户要求记录、项目已有 `spec/` 规范、或任务规模较大时写入 `spec/`。
4. 不编造文件内容、项目结构、运行结果或历史。
5. 最终输出中文（术语/专有名词可保留英文）。

## 与原生 Plan Mode 的协作
- 若任务涉及多文件修改、架构权衡或实现前探索，优先进入原生 Plan Mode。
- 若项目要求把方案沉淀到仓库，再将计划整理为 `spec/04_design.md` 或 `spec/05_plan.md`。
- 若只是轻量建议，不必创建文件。

## 工作流
1. 判断输入是否足够：
   - 若目标、边界、验收标准仍不明确，转回 `my-skill-specify`。
2. 做可访问性判断：
   - 若缺关键上下文且当前无法访问，明确说明“当前无法访问”，一次性列出所需文件或片段。
   - 在现有信息基础上给最小可行建议（MVP），并标注假设与风险。
   - 严禁编造文件内容、目录结构、配置、运行结果或项目历史。
3. 选择输出模式：
   - Design：聚焦方案对比、接口或数据设计、风险与验收。
   - Implementation Plan：聚焦文件变更、实施步骤、测试与迁移。
4. 生成计划：
   - 说明 Why / What。
   - 列出文档变更与代码变更范围。
   - 给出验证方式与状态。
   - 说明风险、兼容性与回滚策略。
5. 按需落盘：
   - 设计偏重时写 `spec/features/<feature-key>/04_design.md`。
   - 实施计划偏重时写 `spec/features/<feature-key>/05_plan.md`。
   - 小任务默认只在对话中输出。
6. 保持文档与测试意识：
   - 计划中应指出需要同步更新的文档。
   - 正式测试放 `tests/`，临时验证脚本放 `scripts/scratch/`。
   - 未运行测试标注 `UNVERIFIED`；失败标注 `BROKEN` 并说明修复方向；通过标注 `VERIFIED` 并附命令与关键结果摘要。

## 切换建议
- 若方案已确认并进入实施或归档阶段，可转入 `my-skill-docs`。
- 若执行中重新暴露关键需求歧义，切回 `my-skill-specify`。
