# 需求澄清（Specify）

将模糊请求收敛为可执行规格，优先减少歧义和返工风险。

## 工作原则
1. 进入澄清语境，只做需求收敛，不直接给完整实现方案或全量实现代码。
2. 不编造未提供的信息；不确定处必须标注“假设”或“待确认”。
3. 默认先在对话中澄清；只有用户明确要求记录，或项目已有固定 `spec/` 规范时，才写入 `spec/`。
4. 若问题可整理为 1-4 个关键选项，优先使用 `AskUserQuestion`；否则分组提出关键问题，避免一次抛出过长问题墙。
5. 最终输出中文（术语/专有名词可保留英文）。

## 工作流
1. 判断是否真的需要澄清：
   - 若目标、范围、输入输出、约束、验收标准已足够清晰，不触发本技能，转入 `my-skill-plan`。
2. 做可访问性判断：
   - 若缺关键文件但当前无法访问，明确说明“当前无法访问”，一次性列出需要补充的文件或片段。
   - 可基于现有信息给最小可行建议（MVP），但必须标注假设与风险。
   - 严禁编造文件内容、目录结构、配置、运行结果或项目历史。
3. 发起采访式澄清：
   - 重点覆盖：目标与范围、用户场景、输入输出、约束、错误处理、验收标准、依赖。
4. 汇总收敛结果：
   - 输出已确认信息、待确认问题、候选方向（可选）、验收草案、建议下一步。
5. 按需记录到 `spec/`：
   - 仅在用户明确要求记录，或项目已有固定规范时写入。
   - 推荐优先写 `spec/features/<feature-key>/01_requirements.md` 与 `03_open_questions.md`。
   - 不要默认一次创建整套结构。

## 标准输出骨架
### 已确认信息
- ...

### 待确认问题
1. ...
2. ...

### 候选方向（可选，标注“待确认/未定稿”）
- 方向 A：...
- 方向 B：...

### 验收草案
- ...

### 建议下一步
- 若关键歧义已消除，转入 `my-skill-plan`。
- 若用户要求正式沉淀，再按需写入 `spec/`。

## 安全与真实性
- 不记录、不输出真实密钥、Token、私钥或生产数据；示例使用占位符。
- 不可声称“已验证”或“已完成实现”，除非实际执行并确认。
