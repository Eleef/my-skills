# 需求澄清（Specify）

## 概览
将模糊需求转为可执行的规格说明，优先消除歧义与缺口，避免因理解偏差导致返工。

## 工作流（按顺序执行）
1. 进入 SPECIFY 语境并遵守边界：
   - 必须输出：关键问题清单（采访式澄清）。
   - 允许输出：1～2 个候选方向/草图（标注“未定稿/待确认”）。
   - 禁止输出：完整定稿方案、全量实现代码、声称已验证。
2. 先做可访问性判断：无法读取的文件必须说明“当前无法访问”，进入降级模式：
   - 一次性列出需要用户补充的文件/片段清单。
   - 在现有信息基础上给出最小可行建议（MVP），并显式标注假设与风险。
   - 严禁编造文件内容、目录结构、配置、运行结果或项目历史。
3. 发起采访式澄清：一次性列出关键问题，覆盖目标范围、用户场景、输入输出、约束、错误处理与验收标准。
4. 允许给出 1～2 个候选方向/草图，但必须标注“待确认/未定稿”。
5. 汇总已确认内容与未决项，给出下一步建议（例如转入 planning）。若需要留档，执行“需求记录”。

## 采访式澄清清单（简版）
- 目标与范围：要解决什么问题？明确“做/不做”。
- 用户与场景：谁使用？如何使用？
- 输入输出：输入格式/来源/规模？输出格式/结构/示例？
- 约束与性能：时延、并发、资源限制、平台约束？
- 错误处理：失败如何表现？是否需要降级与可观测性？
- 验收标准：可测试的 Definition of Done？
- 风险与依赖：依赖哪些系统/文件/权限？

## 输出要求
- 最终对用户输出中文（术语/专有名词可保留英文）。
- 不可编造任何未提供或未验证的信息；不确定处标注假设与风险。
- 不记录、不输出真实密钥/Token/私钥/生产数据；示例使用占位符。

## 需求记录（按需执行）
- 需求工作区固定在 `spec/`：
  1) 优先把“需求确认摘要 + 未决项 + 下一步”写入 `spec/` 下对应功能目录
  2) 推荐路径：`spec/features/<feature-key>/01_requirements.md`、`02_acceptance.md`、`03_open_questions.md`
  3) 若目录或模板不存在，先创建最小结构并在 `spec/00_index.md` 建立索引
  4) 无法写入时明确说明原因并给出需要的文件清单
- 若用户要求阶段性归档到 `docs/`，提示切换到 my-skill-docs 技能完成归档。
