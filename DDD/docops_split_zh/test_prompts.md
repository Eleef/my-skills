# DocOps 技能测试提示词 (Prompts)

使用这些提示词进行轻量级的手动触发测试。这些提示词旨在帮助你判断应选择 `docops-mode` 还是 `doc-template-router`。

## 预期触发 `docops-mode` 的提示词

### 1. 仓库文档重构
帮我梳理这个仓库现有文档的组织方式。我想把平台通用文档和某个 app 的业务文档彻底拆开，同时把 `dev_status` 的维护方式一起定下来。先不要直接写大段正文，先按流程给我方案。

### 2. 调试与文档同步
我刚解决了一个部署问题，现在需要把排障过程、相关风险、以及后续要跟进的事项补到仓库文档里。顺便帮我判断这次是不是还要更新 `active_task` 或 `history_log`。

### 3. 多步骤功能文档编写
我要推进一个新功能，涉及 API、数据字段、验收说明和当前迭代状态。你不要只回答建议，直接按完整 DocOps 工作流带我走。

### 4. 从规范 (Spec) 到实现的桥接
这个需求已经聊清楚了，也有 feature 草案和 API 约束。现在我要开始改代码，你先帮我把实现检查清单 (implementation checklist)、受影响代码面、验证计划和要同步的文档列出来。

## 预期触发 `doc-template-router` 的提示词

### 5. 首先选择模板家族
我要新增一篇文档，但我还没决定它应该是 feature、issue、还是 ADR。你先别写内容，先帮我路由到正确模板。

### 6. 微型编辑 vs 结构化编辑
这个 `api_schema.md` 我只想补两个 endpoint 的说明，但也可能顺手重排目录。你先帮我判断这次到底算微改还是结构改。

### 7. 检查清单驱动的状态更新
我准备更新 `docs/dev_status/history_log.md`，但只想确认这次需要遵守哪些最小检查清单 (checklist)，不需要你先写正文。

### 8. 标准路由
我们最近在几个 app 里都遇到了同样的接口命名和错误返回风格不一致的问题。我不想把这事写进单个 feature 文档，你先帮我判断这应该落到 standards、ADR、还是 dictionary。

## 通常不应触发这些技能的提示词

### 9. 纯代码变更
帮我把这个 TypeScript 函数改成更简洁的写法，不需要动文档。

### 10. 通用解释
解释一下什么是 SSOT，用中文简单说明就行，不结合当前仓库。

## 快速评估指南
- 如果提示词关于仓库文档工作流、状态跟踪或协调的文档变更，首选 `docops-mode`。
- 如果提示词主要关于选择格式、编辑路径、检查清单、模板家族或标准 vs ADR vs 词典路由，首选 `doc-template-router`。
- 如果提示词仅涉及编码或通用解释，这两个技能都不应是必需的。
