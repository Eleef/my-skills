# DocOps 模式 (Mode)

将此技能用于较重的仓库文档工作。此技能是工作流层，而非始终开启的规则层。

## 此技能的作用
- 确定请求是否属于 DocOps 范围。
- 对相关项目文档运行精简的引导 (bootstrap)。
- 切换到正确的工作状态。
- 在任务不明时一次性批量澄清缺失的需求。
- 决定任务是否需要模板路由技能。
- 生成结构化的交付内容和干净的交付笔记。

## 工作流

### 1. 应用范围门控 (Scope gate)
仅在用户请求涉及当前仓库的代码、文档、架构、调试、部署说明或迭代工作流时使用 DocOps 模式。

如果请求明显在项目之外，请正常回答，不要强加 DocOps 规范。

### 2. 选择工作状态 (Working state)
选择一个状态，并通过行为隐式表达：
- `PLANNING`（规划中）：请求仍不明确或不完整。
- `DESIGN`（设计中）：对比方案、契约、风险或验收边界。
- `IMPLEMENTING`（实施中）：进行具体的文档/代码变更。
- `VERIFYING`（验证中）：运行检查或总结验证状态。
- `DONE`（完成）：收尾并更新状态/历史记录。

不要假装同时处于多个状态。保持进度清晰。

`Working state`（工作状态）跟踪工作流阶段。它与 `VERIFIED` / `BROKEN` / `UNVERIFIED` 不同，后者跟踪特定结果的证据状态。

### 3. 运行精简引导 (Compact bootstrap)
对于非琐碎任务，在可用时按此顺序仅读取最少量的相关文件：
1. `docs/00_project_index.md`
2. `docs/dev_status/active_task.md`
3. 如果涉及某个应用，读取 `docs/apps/<app>/00_readme.md`
4. 如果涉及契约/Schema，读取 `docs/apps/<app>/dictionary/*.md`
5. 如果涉及部署或路由，先读取 `docs/apps/<app>/ops/**`，然后是 `docs/platform/ops/**`

如果文件不可用，请进入降级模式，而不是凭空猜测。

### 4. 诚实使用降级模式 (Degraded mode)
如果你无法访问相关文件或输出：
- 明确指出哪些内容不可用。
- 请求提供最小的缺失集合。
- 提供一个 MVP（最小可行产品）答案，并清晰标注假设和风险。
- 严禁虚构仓库事实。

### 5. 批量处理不明确的问题
如果任务不明确，请提出一组集中的澄清问题，而不是进行多次琐碎的一问一答。

重点关注：
- 目标 (Objective)
- 范围边界 (Scope boundaries)
- 目标应用或平台区域
- 验收标准 (Acceptance criteria)
- `docs/dev_status/**` 所需的轨迹 (Track)

在得到回答之前，你可以提供草案，但必须标注假设。

### 6. 尊重应用/平台分离
- 应用特定的内容属于 `docs/apps/<app>/**`。
- 共享的、跨应用的指南属于 `docs/platform/**`。
- 保持平台文档的通用性，并链接到应用文档，而不是吸收应用特定的行为。

### 7. 执行 SSOT 行为
- 应用词典 (Dictionary) 文件是应用作用域行为的契约权威。
- 平台拥有的共享契约可能存在于 `docs/platform/<component>/specs/**` 中。
- 功能文档负责叙述，并链接到相关的权威文档，而不是重复陈述。
- 不要在功能或运维 (Ops) 文档中维护重复的 Schema。
- 如果任务改变了行为，请同时更新实现和受影响的文档。

### 8. 在需要时委派模板工作
如果任务需要模板路由、结构化文档生成或检查清单驱动的起草，请使用兄弟技能 `doc-template-router`。

特别是在以下情况使用：
- 创建新文档
- 对现有文档进行大规模重构
- 在功能 / 词典 / 标准 / 开发状态 / 问题 / ADR 格式之间进行选择
- 决定是微型编辑还是结构化编辑流程

### 9. 明确桥接规范到实现
在代码变更开始前，使桥接过程可见。

将以下项作为精简的实现入口检查清单 (Implementation entry checklist) 记录：
- 正在解决的需求或用户目标
- 定义变更的主要规范文档
- 适用的标准、ADR 或平台约束
- 预期变更的代码区域
- 必须原子更新的文档
- 必须验证的验收标准
- 验证计划：命令、冒烟测试或可观察的结果

使用这些桥接规则：
- 如果行为、API 或数据契约发生变更，请在代码变更之前或与之同步更新相关的权威规范：
  - 应用拥有的契约变更 -> `docs/apps/<app>/dictionary/**`
  - 平台拥有的共享契约变更 -> `docs/platform/<component>/specs/**`
- 如果变更创建了可重用的跨应用指南，将其路由到 `platform/standards/**`，而不是隐藏在功能文档中。
- 将验收标准视为验证的源头，而非装饰性文字。
- 如果实施过程中发现了规范缺口，请在声称完成前修补规范。

### 10. 搁置尚未准备好实施的想法
如果用户说“现在不要”、“以后再说”，或者分享了一个粗略的想法但还不想执行：
- 为简单的未来工作在 `docs/dev_status/todo.md` 中添加一个简洁项。
- 为较大的想法创建一个草案设计或标准笔记，然后从 todo 中链接它。
- 保持笔记简短、明确且标记轨迹。

### 11. 交付格式
对于 DESIGN / IMPLEMENTING / VERIFYING 输出，使用此结构：
1. 当前工作状态 (Current Working State)
2. 摘要 (Summary)
3. 文档变更 (Docs changes)
4. 代码变更 (Code changes)
5. 验证 (Verification)
6. 风险与回滚 (Risks and rollback)

使用明确的 `VERIFIED`、`BROKEN` 或 `UNVERIFIED` 标签。

**简短示例：**
- 当前工作状态：`VERIFYING`
- 验证：
  - 冒烟测试：`VERIFIED`
  - 生产环境一致性检查：`UNVERIFIED`

### 12. 完成与暂停处理
如果任务中途暂停：
- 仅更新 `active_task.md` 中相关的轨迹块。
- 不要覆盖无关的轨迹。
- 保留目标、当前焦点和后续行动。

如果任务已完成并确认：
- 在 `history_log.md` 顶部添加一条简洁的摘要。
- 重置或更新该轨迹的活动状态。
- 保持每个历史条目为单轨迹。

## 实用启发式方法
- 优先选择任务所需的最少文档覆盖面。
- 保持过程可见，但输出要精简。
- 将 DocOps 视为工作流支架，而不是过度文档化的理由。
- 如果任务只是极小的单章节更新，跳过沉重的工作流，进行符合规范的最小编辑。
- 如果用户的主要需求是模板选择、结构化路由或基于检查清单的起草，尽早移交给 `doc-template-router`，而不是重复它的工作。

## 示例触发场景
- "帮我梳理这个仓库的 docs 结构，然后给我一个落地改造方案。"
- "把这个功能的文档、dev_status、history 一起理顺。"
- "这个排障过程需要补到 docs 里，你按规范带我走完整流程。"
- "我不确定这块应该写到 platform 还是 app 文档，你来判断并落地。"
- "这个需求已经有 spec 了，我现在要开始改代码，你先帮我做实现前桥接和验证计划。"

## 辅助文件
- 如果你需要快速温习，使用 `references/bootstrap_and_states.md` 获取精简的状态/引导速查表。
- 当任务从需求/规范转向代码和验证时，使用 `references/implementation_bridge.md`。
